1. Overview

Line breaking is the process by which line breaking opportunities are determined in a given line of text. In Western text, those opportunities are based either on 'space/white space' or hyphenation. In many East Asian writing systems, line-breaking opportunities exist almost anywhere. The forbidden cases are specified as 'Kinsoku' in Japanese and can be extrapolated to a Chinese context. For example it is not desirable to have a closing parenthesis ')' starting a line. Korean rules are somewhere between Western and Japanese rules as Korean texts typically use space as a word delimiter. Determining line breaking opportunities for Asian writing systems usually requires a two character context, possibly three if the decision to whether breaking across space or not is also analyzed. (It may be seem obvious that we should always break across space, but there are cases in Japanese typography where it is not desirable to do so).

The following context elements should be considered:

  • It may be desired to just provide Western type line breaking (i.e. on space characters and related 'white' space characters. In that case many Asian writing systems (Japanese, Chinese) will behave poorly, but it is very fast. This requires the minimum context analysis.
  • Provide normal Kinsoku rules. This is the standard mode where symbols are allowed to either start or end a line based on their relationship to previous or following characters. In that case most ideographic characters are allowed to either start or end a line.
  • Apply stricter Kinsoku rules by not allowing small kana characters to start a line.
  • Thai line breaking is performed using a contextual dictionary lookup. This specification assumes that a special module provides the following information (Last character of a Thai word, First character of a Thai Word, middle Thai letter) from a Thai string.

Two additional contextual elements apply to the previous rules:

  • Allow Western text (really non-East Asian text) to break everywhere. This may provide incorrect line breaking within Western text but is more visually pleasing in Asian context.
  • Keep ideographic characters and Hangul together. This mode is used in Korean context where typically Hanja and Hangul appear in clusters separated by space characters.

Finally, some writing systems supported by Unicode 2.0 are not yet supported in this specification:

  • Lao. It requires a dictionary look up similar to Thai. As it is not yet available, it is processed as a default 'Alpha' script. This really means that, in the absence of space characters or <BR> tag, each Lao paragraph will be displayed as a single line.
  • Tibetan. It requires handling of the syllable separator 'bar thseg' (ref The World's Writing Systems, Oxford, page 435) and the understanding of its implementation in Unicode. The default behavior is the same as for Lao.

Other writing systems planned for Unicode 3.0 have been already included, like Burmese, Khmer, Thaaana, etc... It is expected that most of them should be able to be supported by the current logic as most of them use the space character as word delimiter.

2. CSS properties

(Extract from CSS Intl document, subject to change), it is also possible to look at: Proposal for Intl CSS extension

line-break property

Value: normal | strict
Initial: normal
Applies to: all elements
Inherited: yes
Percentage values: N/A

This property selects the line breaking level. Possible values:

  • normal - line breaking opportunities are determined by the Latin, Thai and regular ideographic line breaking rules with an additional commonly used set of restrictions.
  • strict - line breaking opportunities are determined by the Latin, Thai and regular ideographic line breaking rules with an additional larger set of restrictions.

word-break property

Value: normal | break-all | keep-all
Initial: normal
Applies to: block elements
Inherited: yes
Percentage values: N/A

This property controls line-breaking behavior inside of words. Possible values:

  • normal: Keep all non Asian scripts (according to their own rules), CJK and Hangul break everywhere or according to the rules of line-break mode (kinsoku).
  • break-all: same as normal for CJK and Hangul, but now non Asian scripts can also break anywhere. This options is used mostly in context where the text is predominantly using CJK characters with few non Asian excerpts and it desired that the text be better distributed on each line.
  • keep-all: same as normal for all non Asian scripts. CJK and Hangul are kept together. This option should only be used in the context of CJK used in small clusters like in the Korean writing system.

For convenience, the word-break property can also expressed using two boolean flags combining word-break values:(not part of the CSS specification):

  • break-latin
    • on when word-break = break-all
    • off when word-break = normal | keep-all
  • break-CJK
    • on when word-break = normal | break-all
    • off when word-break = keep-all

The table driven approach used later in the document is designed to use these two flags directly.

3. JIS Line breaking

The JIS 4051:1995 specification specifies a line-breaking behavior. It is based on the type of the previous and the next characters. In addition, it assumes a thorough knowledge of the contextual environment (e.g. character part of a unit range). Furthermore, it describes behavior between line objects like Ruby, inline notes, attachment, etc. Therefore, it is not directly applicable to a model that has a limited contextual awareness like ours.

Another major difference concerns the processing of the narrow space character. In JIS, it has its own class. In Trident, the narrow space is not part of the classification. 4. Mapping between Partitions and Line breaking behavior

Simply put, the line-breaking problem can be expressed as determining line-breaking opportunities within a three-character context where the middle character is predetermined (space) while taking into consideration the current values of the line-break and word-break properties. The solution is as following:

  1. Map partitions to line-breaking behavior classes, based on character code and two additional flags:
    1. Narrow or Wide, based on used font
    2. Kinsoku strict or not (change the mapping for some Kana characters)
  2. Use a 2D line breaking behavior table, one direction for preceding character, the other for following character
  3. In the table, allow these behaviors:
    1. line breaking possible
    2. no line breaking even across space
    3. no line breaking but can break across space

Thai line breaking

The Thai language is one of several languages that do not require white space between words. And it is not possible through simple character classification to determine line breaking. Therefore for Thai, the context around the character need to be analyzed to determine breaking opportunities. Note however that this is not necessary for the Thai currency symbol and the Thai digits that are classified without dictionary. Once done, the result is fed back into the Kinsoku logic to avoid separate path and also to make easier to process boundary cases. The following example shows how the logic is implemented:

Assuming that in a string called 'LabcdefM', 'L' and 'M' are Latin characters, 'abcded' are Thai characters with a breaking opprtunity between 'c' and 'd', two breaking classes are provided for each character:

brkclsAfter

brkclsBefore
L Alpha Alpha
a ThaiMiddle ThaiFirst
b ThaiMiddle ThaiMiddle
c ThaiLast ThaiMiddle
d ThaiMiddle ThaiFirst
e ThaiMiddle ThaiMiddle
f ThaiLast ThaiMiddle
M Alpha Alpha

And the following classes are used for the breaking opportunity analysis as input in the line breaking behavior table:

First character (before) Second character (After)
{L, a} Alpha ThaiFirst
{a, b} ThaiMiddle ThaiMiddle
{b, c} ThaiMiddle ThaiMiddle
{c,d} ThaiLast ThaiFirst
{d, e} ThaiMiddle ThaiMiddle
{e, f} ThaiMiddle ThaiMiddle
{f, M} ThaiLast Alpha

It should be noted that with this logic, the line 17 (Thai first) and the column 18 (Thai Last) should never be used..

7. LS Line Breaking implementation notes

Line services 3.0 implements Line Breaking using these APIs and flags:

  1. LsSetBreaking() in which IE5 provides a two dimensional array defining class behavior for character pairs. The behavior is described by two flags:
    fCanBreakAcrossSpace,
    fCanBreakWithoutSpace.
    Which flags to set is determined by the index value described above (1 to 4).
  2. Another flag called fFmiWrapAllSpaces is TRUE for TEXTAREA and makes the AsciiSpace Row and Column active. Basically, it makes the space character part of the standard line breaking logic, instead of being special cased, making the break across space or not irrelevent.
  3. pfnGetBreakingClass() which is a callback which fetches a breaking behavior class for a given character used a leading or following characters in a character pair. The class values are bytes that are used as index in the class behavior array specified above. The current implementation only supports one breaking class behavior class per character, it doesn't matter whether the character is a leading or following one in a pair.
  4. fFmiApplyBreakingRules is a LS PAP flag indicating whether or not the FE Line breaking behavior should be activated. It is always on.
  5. pfnCanBreakBeforeChar() and pfnCanBreakAfterChar() returns to LS the rules expressed for line breaking rules around inline objects.

LS maintains a structure called LSTXTCFG which contains special characters with an implied behavior. When those characters are active (slot occupied with a value different from 'chUndef') they participate in the line formatting prior to be provided to call back functions or used in behavior tables. For example, most of these characters are line breaking opportunities by the mere fact of being included in that structure. A tentative list of these characters is:

wchSpace, wchHyphen, wchTab, wchEndPara1, wchEndLinePara, wchColumnBreak, wchPageBreak, wchNonReqHyphen, wchEmDash, wchEnDash, wchEmSpace, wchEnSpace, wchOptBreak.

This supercedes the behavior described by the previous tables. It is sometimes incomplete (for example, it does not include other spaces and hyphens defined by Unicode) or in contradiction with JIS requirements (EmDash is an inseparable character in JIS and therefore is not an unconditional line breaking opportunity). Thai word breaking

In order to find word break opportunities for breaking lines and navigating to word boundaries, we must analyse the content of the string. We use UniScribe (USP10.DLL) to perform this task.

The task of analysing a Thai type string we extract a string with up to 48 characters before the current character position and up to 26 characters after the current position. The rationale for this is:

  1. The maximum average word length for Thai is 12 characters. To get an accurate analysis we need to have at least four words to get a start of word boundary.
  2. The longest Thai word is about 20 characters. When navigating from word to word we want to make sure we know where the next word begins if we are in the first position of the longest word.
  3. We move in either direction until we find the first non-Thai type character.

Once the string is created we use ScriptItemize. Then we use ScriptBreak on the chunk of text in which our current character position is located. ScriptBreak returns an array of information about each character. The information includes if it is a character or word boundary.

For line breaking, we pass back the class of the current character for its break opportunity before and after the character. There are four possible combinations:

After Before
brkclsThaiLast brkclsThaiFirst
brkclsThaiMiddle  brkclsThaiMiddle

It is only legal to break Thai type words after brkclsThaiLast and before brkclsThaiFirst.
