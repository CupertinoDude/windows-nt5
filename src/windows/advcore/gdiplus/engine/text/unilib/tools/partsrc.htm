Abstract

This document describes a Unicode partitioning scheme aiming at providing a fully deterministic behavior description for groups of Unicode characters. When analyzing typographical behaviors such as line breaking, expansion, compression, etc.. it is clear that characters can be grouped together. For example, all 'open parenthesis characters' share typically the same behavior for line breaking (i.e. it is desirable to avoid ending a line with such a character). When comparing the various typographic processes it becomes quickly obvious that large groups of characters share the same behaviors. Therefore, instead of using large property tables for each process, a simple Unicode space partitioning can be used as the foundation to describe all of them.

Currently the metrics are as follows: the Unicode characters (about 35 00 of them) are grouped into about 200 partitions, these partitions are mapped into 5 behavior arrays (font determination, line breaking, word breaking, modwidth and ruby overhanging). Each of these arrays has between 15 and 25 index values. The resulting indexes are finally used in 1D or 2D array which fully describes typographical behavior. It is in essence a table driven approach with no platform dependency.

1. Character classification

Recently Japan has come up with a typography standard (JIS 4051:1995) which specifies in great details the rules concerning line breaking, justification and character classifications. As the base character encoding is Unicode (referred to in the standard as JIS X221 which is the Japanese transposition of ISO/IEC 10646:1993 part 1), these rules can easily extended to accommodate other FE languages.

It is worth to mention that, while JIS 4051 specifies a character classification (page 23 and 35), these classes do not have exclusive contents, i.e. characters may belong to several classes. This makes the implementation rather difficult. To solve that issue, the original JIS classification can be broken into two separate concepts:

  1. A partitioning of the Unicode code space that does not allow characters to be part of several partitions at the same time.
  2. Class behavior specifications that allow a single character belonging to a specific Unicode partition to behave differently depending upon the context. All characters of the same partition can be processed following the same behavior determination rule. These class behaviors are dependent upon the typographical process (line breaking, character compression, auto space,etc.).

Furthermore JIS deals with half and full width characters in a way that is not compatible with industry practice. JIS is assuming that the Basic Latin part of Unicode behaves like full width character in a Japanese context. This is simply not true in a Windows environment. Therefore, when following the JIS treatment of an ASCII character with width dependencies, this specification uses the full width variant instead. There are also cases where half width, full width and ambiguous width (i.e. context dependent width) must be considered.

The intent of the following specification is to cover JIS functionality at the user level but with the character encoding practice of Windows 95 and Windows NT as used in Japan and in other countries.

3. Unicode Partitions definition

This section defines a partition of the Unicode code space based on properties relevant to FE and Complex Script (Bidi and South Asian) typography. The partitions are specified as collections of characters that share the same process for determining a typographical behavior (like line breaking, CSC and compression/expansion). A character belongs only to a partition, i.e. partitions do not intersect. Partitions belong to four major types concerning the character width of their content (explicitly narrow, explictly half width, explicitly wide, or ambiguous, i.e. the width is determined by context). The mapping from partition to class behavior depends upon the kind of class (there is a mapping for line breaking, another one for CSC, etc.) and each kind of class has its own behavior array. This mapping may depend on context analysis or on the formatting options on font selected.

Wide (W) characters are characters that are usually displayed with an Em width character width before adjustment. Narrow characters are either half width characters (H) displayed with ½ Em and part of a FE font or characters usually not part of a FE font (N), typically the later characters have no predetermined width. Ambiguous width (A) characters are characters that can have either width depending upon the context.

An important aspect of the partitioning is the fact that all characters in each partition share a common behavior concerning the possible removal of blank space within a cell. For example, if a character belongs to the ‘Opening Parenthesis and Quotes’ Wide form (W) it is expected that the left half of its bounding box is blank. If this is not the case, the deviant characters need to have their own partition. If a character is ambiguous concerning its width, it is however expected that it will share the wide aspect ratio when its width is determined to be wide based upon context.

From the foregoing, it can be seen that the original JIS classification is different from the one defined here. However, the essential point is that the 20 JIS character classifications are all available as specific behaviors. 4. Unicode Partitions enumeration

Following are the Unicode Partition definitions, Sources include JIS-X-4051:1995 and The Unicode Standard Version 2.0. A major source was also the Unicode mapping tables for code page 932 (Japan), 936 (Simplified Chinese), 949 (Korean) and 950 (Traditional Chinese). These tables were inverted and merged for all Unicode characters below 0x3000 to provide input for the ambiguous characters. In addition each partition entry indicates behaviors associated with the partition. The syntax is as follows:

  • Font expresses the font behavior when the characters must be connected to a script font.
  • LB describes the Line Breaking class behavior of these characters.
  • WB describes the Word Breaking class behavior of these characters.
  • MW describes the Mod Width class behavior of these characters.
  • OH describes the Ruby Overhanging class behavior of these characters.
  • Ranges are expressed either as single 4 hex value followed by tab, or as a pair of 4 hex values separated by a hyphen-minus and followed by a tab. The following name is only for easy reference.

Their values is not always current  (it is meant as a hint for partition usage) as the actual mapping is maintained in the various specifications describing these behaviors. For latest see:
